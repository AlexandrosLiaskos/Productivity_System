I have my own custom `system` of *long-term productivity logging*. This is because I am a professional and thus I have to be very organized in the way I work in order to ensure *reliability, scalability, reusability, storage, etc*. In the system, everything is an `entry` to the system that has a `date-stamp` (date: *YYYYMMDD*). All entries are *files* with *filenames ending in the date-stamp*. Entries are organized *only to one higher level of hierarchy*, which are the `projects` with the relationship of *1:many* (This is in order to tackle one of the issue of information organization which relates to *nesting complexity/chaos*). The *entries enums/types* are the `tasks`, `logs` and `notes`. The *tasks enums/types* are the  `in progress`, `queued`, `on hold`, `conpleted` and `canceled` (Showcasing the *state of the task*, while the tasks are meant for *`action`/s within a project that need to be implemneted* and *may have a `deadline`*).  Logs on the other hand showcase *what has been implmented and loged* (important for the future for *reusability of knowledge, information, workflow/s, source code, etc*, while also for *security of what has been done and when*. Notes function as *fleeting notes* that are being kept for long-term memory and are meant to be handled in the future. The handling of notes is the action of converting a note to a task/s (This functions by referencing to the new task, the note that it originated from). All entries are loged as `JSON` files, with the exception of notes which are `markdown` files. All entry files have `IDs` *their filenames*.  This filesystem ensures a *traceable information and history/progress search/indexing*. For data protection, the systen is *git-tracked* and *hosted/synced to a github project* (cloud environment), while the projects have the reference in their `metadata` to the *github project url containing the project data/code if existing*. Keeping all information in *files and separate* and the *project's source-code/data and the logging system's data of the same project separate* , ensures *portability, editability, compatibility, and separation of concerns*.
